# InferHub retrieval eval harness

A small console tool that measures retrieval quality on **your** corpus. It runs a golden set of
queries against a live coordinator in each mode — `vector`, `keyword`, `hybrid`, `hybrid+rerank` —
and prints **Recall@k**, **MRR**, **nDCG@k** and **median latency** as a table.

It exists because "hybrid search improved retrieval" is an empirical claim. If you can't measure it,
don't put it in a slide. This tool is how you find out whether hybrid or reranking helps on the data
you actually have — and sometimes the answer is "not much", which is worth knowing before you turn
either on.

It is **not** built into the coordinator or node images. It is something you run, not something you
deploy.

## Build & run

```bash
dotnet run --project tools/InferHub.Eval -- \
  --base-url http://localhost:5080 \
  --api-key  $INFERHUB_API_KEY \
  --collection handbook \
  --golden    ./golden.jsonl \
  --k 5
```

The API key can also come from the `INFERHUB_API_KEY` environment variable. It needs a **client**
key (the same scope that ingests and queries), not an admin key.

Output:

```
mode              Recall@5       MRR    nDCG@5   median ms
--------------------------------------------------------------
vector               0.612     0.540     0.571        41.0
keyword              0.548     0.611     0.560        22.5
hybrid               0.744     0.690     0.712        58.0
hybrid+rerank        0.771     0.742     0.749       690.0
```

(Those numbers are an illustration of the shape of the output, not a result. Run it on your corpus.)

## The golden set

One JSON object per line (JSONL):

```json
{"query": "What does error E-4021 mean?", "relevant": ["handbook#12", "6a5f...chunkid"]}
{"query": "parental leave policy",        "relevant": ["handbook-v2"]}
```

- `query` — the user's question, in their words.
- `relevant` — identifiers of the chunks that *should* come back. Each entry is matched against a
  retrieved hit's **chunk id** *or* its **documentId**, so you can write the golden set at whichever
  granularity you have to hand. `GET /api/collections/{c}/documents` and `.../{id}/chunks` give you
  both.

### One warning, and it matters

**A golden set generated by the same model you are about to evaluate is not evidence. It is a
mirror.** If you ask an LLM to invent questions from your documents and then measure how well
retrieval answers those questions, you are measuring how self-consistent the model is, not how good
retrieval is. Build the golden set from **real questions people actually asked** — support tickets,
search logs, the questions in your inbox — or don't build it at all. A small honest golden set beats
a large synthetic one.

## What the numbers mean

- **Recall@k** — of the chunks that should have come back, how many did, within the top `k`.
- **MRR** — mean reciprocal rank of the *first* relevant chunk. Rewards putting a right answer high.
- **nDCG@k** — rank-weighted relevance, normalised so 1.0 is the ideal ordering.
- **median ms** — per-query wall time including the network hop. Reranking adds a fleet round trip;
  this is where you see its cost.

If hybrid does not beat vector on your corpus, the honest thing is to say so — and to leave the
default at `vector`, which is exactly why that is the default.
